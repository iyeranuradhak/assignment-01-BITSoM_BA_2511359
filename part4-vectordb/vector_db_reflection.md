Vector DB Use Case

A traditional keyword-based database search would not be sufficient for a system that allows lawyers to query large contracts using plain English questions. Keyword search relies on exact or partial word matching, which means it may fail when the query uses different terminology than the document. For example, a lawyer searching for “termination clauses” might miss relevant sections labeled as “contract cancellation,” “end of agreement,” or “exit provisions.” This limitation makes keyword search unreliable for complex legal documents where language can vary significantly.

In contrast, a vector database enables semantic search by understanding the meaning of the text rather than just matching words. Using embeddings generated by models such as sentence-transformers, both the contract text and the user’s query are converted into numerical vectors in a high-dimensional space. Similar meanings result in vectors that are close to each other, even if the wording is different.

In this system, the 500-page contracts would first be broken into smaller chunks (e.g., paragraphs or sections), and embeddings would be stored in a vector database. When a lawyer submits a query, it is also converted into an embedding, and the system retrieves the most semantically similar chunks using similarity search.

This approach allows lawyers to find relevant information quickly and accurately, regardless of phrasing differences. Therefore, a vector database is essential for enabling efficient, intelligent search in large, complex legal documents.
